## 2026-04-02 - Tenant Isolation IDOR in Admin API\n**Vulnerability:** Admin endpoints for managing shifts (`/api/admin/shifts/route.ts`) verified the user was an admin for the `serverId` provided in the request, but failed to verify that the target `shiftId` actually belonged to that `serverId`. This allowed an admin of Server A to end or delete shifts from Server B by supplying Server A's `serverId` but Server B's `shiftId`.\n**Learning:** Checking permissions against the request parameters is insufficient. The system must verify the relationship between the target object and the authorized context.\n**Prevention:** When modifying or deleting a resource by ID in a multi-tenant system, always explicitly verify that the retrieved resource belongs to the authorized tenant context (e.g., `if (shift.serverId !== serverId) return Forbidden`).
